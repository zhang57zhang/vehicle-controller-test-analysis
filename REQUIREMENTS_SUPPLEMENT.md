## 1. 车载测试数据格式全景图

### 1.1 测试阶段与数据格式矩阵

| 测试阶段 | 主要数据格式 | 典型文件扩展名 | 数据特点 | 采样频率 | 分析重点 |

## 2. 数据规模与时间同步

### 2.1 数据规模分析

| 数据类型 | 典型大小 | 说明 |
|---------|---------|------|
| **单次测试数据** | 2KB ~ 500MB | 跨度大，需要分级处理 |
| **小型测试** | 2KB ~ 1MB | 手动测试结果、简短日志 |
| **中型测试** | 1MB ~ 50MB | 标准HIL测试、功能测试 |
| **大型测试** | 50MB ~ 200MB | 长时域测试、耐久测试 |
| **超大型测试** | 200MB ~ 500MB | 整车测试路采、故障复现 |

### 2.2 多数据源时间同步机制

#### 2.2.1 时间同步挑战

```
场景示例：一次HIL测试可能同时产生
1. CAN总线数据 (100Hz, 时间戳基于CANoe)
2. 模拟量采集数据 (1kHz, 时间戳基于DAQ硬件)
3. 数字量记录 (10kHz, 时间戳基于FPGA)
4. 故障注入日志 (事件驱动, 无固定频率)

问题：
- 不同设备时间基准不同
- 采样频率不一致
- 时钟漂移和延迟
- 触发信号对齐问题
```

#### 2.2.2 时间同步解决方案

**方案A: 基于公共时间基准**
```
1. 所有数据源使用统一时间基准（如GPS/PTP）
2. 记录每个数据源的时间偏移
3. 导入时进行时间对齐
4. 插值处理频率不一致的数据
```

**方案B: 基于触发信号**
```
1. 使用触发信号作为时间对齐标记
2. 记录触发事件在不同数据源的时间戳
3. 计算时间偏移并应用
```

**方案C: 后处理对齐**
```
1. 允许各数据源独立时间戳
2. 导入后基于公共事件对齐
3. 提供手动时间调整接口
4. 支持多种插值方法（线性、样条等）
```

**建议**: 结合方案A和C，优先使用公共时间基准，提供后处理对齐能力

#### 2.2.3 数据对齐实现要点

```python
# 数据对齐核心逻辑伪代码
class DataSynchronizer:
    def align_data_sources(self, sources):
        # 1. 提取时间戳
        timestamps = {source: source.get_timestamps() for source in sources}

        # 2. 计算公共时间范围
        common_start = max([ts.min() for ts in timestamps.values()])
        common_end = min([ts.max() for ts in timestamps.values()])

        # 3. 确定目标采样率（最高频率的公倍数或用户指定）
        target_rate = self.calculate_target_rate(sources)

        # 4. 生成目标时间序列
        target_time = np.arange(common_start, common_end, 1/target_rate)

        # 5. 对每个数据源进行重采样
        aligned_data = {}
        for source in sources:
            aligned_data[source] = self.resample(
                source.data,
                source.timestamps,
                target_time,
                method=self.get_interpolation_method(source)
            )

        return aligned_data
```

#### 2.2.4 时间同步元数据

需要记录以下元数据以支持追溯：
- 原始时间戳（raw_timestamp）
- 对齐后时间戳（aligned_timestamp）
- 时间偏移量（time_offset）
- 插值方法（interpolation_method）
- 数据源设备信息（source_device）
- 同步质量指标（sync_quality）

## 3. 核心分析指标体系

### 3.1 指标来源分析

#### 3.1.1 功能测试指标
```
来源：
- 测试用例描述 (Test Case Description)
- DBC 文件 (Signal Definitions)
- 通信矩阵 (Communication Matrix)

指标类型：
1. 信号范围检查
   - 信号值是否在 [Min, Max] 范围内
   - 示例：Battery_Voltage 应在 300V ~ 450V

2. 信号逻辑检查
   - 信号之间逻辑关系是否正确
   - 示例：当 Gear=P 时，Speed 必须接近 0

3. 值表检查
   - 信号值是否在合法值表中
   - 示例：Gear_State 必须在 {P, R, N, D} 中

4. 时序检查
   - 信号变化顺序和时序是否符合预期
   - 示例：收到 Shutdown 指令后，Motor_Speed 应在 2s 内降为 0

5. 抖动和稳定性检查
   - 稳态信号是否稳定
   - 示例：Cruise_Control_Target 100km/h 时，实际速度应在 99~101 km/h
```

#### 3.1.2 性能测试指标
```
来源：
- 测试用例描述 (Test Case Description)
- 控制需求文档 (Control Requirements)
- 性能规范 (Performance Specifications)

指标类型：
1. 响应时间指标
   - 阶跃响应时间
   - 跟随响应时间
   - 计算：从信号变化到目标信号响应的时间

2. 超调量和调节时间
   - 超调量：(最大值 - 目标值) / 目标值 * 100%
   - 调节时间：进入并保持在误差范围的时间

3. 稳态误差
   - 稳态时实际值与目标值的偏差

4. 频域指标
   - 幅频特性
   - 相频特性
   - 截止频率
   - 相位裕度

5. 效率指标
   - 系统效率
   - 损耗分析

6. 温度指标
   - 温升速率
   - 最高温度
   - 热平衡温度

7. 多数据计算指标
   - 多个信号组合计算的结果
   - 示例：Motor_Power = Motor_Torque * Motor_Speed / 9550

8. 时段计算指标
   - 一段时间内的统计指标
   - 示例：平均能耗、最大温度、温度标准差
```

## 4. 测试报告双报告机制

### 4.1 双报告设计理念

```
报告A: 标准模板报告（对外）
├─ 目的：正式交付、存档、对外发布
├─ 内容：精简、专业、格式规范
├─ 特点：数据经过处理，只展示关键信息
└─ 示例：向管理层汇报、向客户交付

报告B: 数据溯源报告（内部审核）
├─ 目的：工程师深度审核、问题追溯
├─ 内容：完整、详细、数据来源标注
├─ 特点：每个数据点都有原始出处
└─ 示例：技术评审、问题分析、审计
```

```
- 单位（unit）
- 计算公式（if computed）
```

#### 报告结构
```
1. 数据元信息
   - 测试基本信息
   - 数据源清单（所有导入的文件）
   - 时间同步说明
   - 数据质量评估

2. 信号溯源表
   - 信号ID
   - 信号名称
   - DBC定义（如果有）
   - 原始信号名（如果经过映射）
   - 单位（原始单位/显示单位）
   - 采样率
   - 数据质量评分
   - 备注（缺失值、异常值说明）

3. 指标计算溯源
   - 指标ID
   - 指标名称
   - 计算公式
   - 输入信号清单（带溯源）
   - 计算时间范围
   - 计算方法
   - 结果值
   - 阈值判断依据

4. 异常事件溯源
   - 事件ID
   - 事件描述
   - 触发条件
   - 触发时间
   - 相关信号清单
   - 原始数据片段（时间窗口）

5. 原始数据引用
   - 完整的原始数据路径
   - 文件哈希值（验证完整性）
   - 数据版本
```

## 5. 本地部署架构

### 5.1 本地部署要求

```
部署约束：
- 数据不出本地网络
- 所有计算在本地完成
- 无云端依赖
- 支持离线运行

安全要求：
- 内网部署
- 数据加密存储
- 访问控制
- 操作审计
```

### 5.2 本地部署架构

```
┌─────────────────────────────────────────────┐
│           用户访问层                         │
│  (Web浏览器 / 内网PC / 工作站)               │
└─────────────────────────────────────────────┘
                      ↓
┌─────────────────────────────────────────────┐
│           Web服务器 (Nginx/Apache)           │
└─────────────────────────────────────────────┘
                      ↓
┌─────────────────────────────────────────────┐
│           应用服务器                          │
│  ┌─────────────────────────────────────┐   │
│  │  Web前端 (React SPA)                │   │
│  └─────────────────────────────────────┘   │
│  ┌─────────────────────────────────────┐   │
│  │  后端API (FastAPI + Gunicorn)      │   │
│  └─────────────────────────────────────┘   │
│  ┌─────────────────────────────────────┐   │
│  │  异步任务队列 (Celery + Redis)     │   │
│  └─────────────────────────────────────┘   │
└─────────────────────────────────────────────┘
                      ↓
┌─────────────────────────────────────────────┐
│           数据存储层                         │

│  │  TimescaleDB (时序数据)             │   │
│  └─────────────────────────────────────┘   │
│  ┌─────────────────────────────────────┐   │
│  │  文件存储 (测试数据文件)            │   │
│  └─────────────────────────────────────┘   │
└─────────────────────────────────────────────┘
```

### 5.3 本地部署优势

```
✅ 数据安全：
   - 无云端传输风险
   - 内网隔离保护
   - 符合企业安全策略

✅ 网络独立性：
   - 不依赖外网连接
   - 低延迟访问
   - 可控的网络带宽

✅ 合规性：
   - 满足数据出境法规
   - 符合行业监管要求
   - 便于审计和追溯

✅ 成本控制：
   - 无云服务订阅费
   - 硬件一次性投入
   - 长期成本更低
```

### 5.4 部署方式

#### 方式A: Docker Compose（推荐）
```
优点：
- 一键部署
- 环境一致
- 易于维护
- 支持多容器编排

步骤：
1. 准备服务器（推荐配置：16GB RAM, 4核CPU, 1TB SSD）
2. 安装Docker和Docker Compose
3. 解压部署包
4. 执行 docker-compose up -d
5. 访问 http://server-ip:8080
```

#### 方式B: 手动部署
```
优点：
- 更灵活的配置
- 性能优化空间大

步骤：
1. 安装Python和Node.js运行时
2. 安装PostgreSQL和TimescaleDB
3. 配置环境变量
4. 启动后端服务
5. 启动前端服务
6. 配置反向代理
```

#### 方式C: 虚拟机镜像
```
优点：
- 开箱即用
- 快速部署

步骤：
1. 下载VM镜像（VMware/VirtualBox格式）
2. 导入虚拟机
3. 启动虚拟机
4. 访问 http://vm-ip
```
